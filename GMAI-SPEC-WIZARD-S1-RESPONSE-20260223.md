## 2. Standardized Subtopic Definitions

### Setup Tab

| ID | Title | Required | Description |
|----|-------|----------|-------------|
| `components` | What's in the Box | **YES** | Complete component list with exact counts. Every piece, card, token, die, board in the box. |
| `board-layout` | Board Layout | NO | How to assemble the play area. Omit for card-only games (Sushi Go, Love Letter, Coup, etc.) or games where there is no central play area to assemble. |
| `player-setup` | Player Setup | **YES** | What each player receives and does before the first turn: pieces, cards, starting resources, player board configuration. |
| `starting-conditions` | Starting the Game | **YES** | Everything between "setup is done" and "the first full turn begins": first-player selection, initial placement/drafting, opening hand draw, starting resources distributed. |

### Rules Tab

| ID | Title | Required | Description |
|----|-------|----------|-------------|
| `turn-structure` | Turn Structure | **YES** | The phases of a turn or round, in chronological order. Number them. This is the skeleton that everything else hangs on. |
| `actions` | Actions & Options | **YES** | All actions a player can take on their turn. Include costs, restrictions, limits, and outcomes. Be exhaustive — this is where "can I do X?" questions get answered. |
| `special-mechanics` | Special Mechanics | NO | Game-specific mechanics that don't fit cleanly into turn structure or actions. Examples: the robber in Catan, hidden roles in Werewolf, combat resolution in Root, the traitor mechanic in Dead of Winter. Omit for simple games with no special systems. |
| `endgame` | Endgame & Scoring | **YES** | What triggers the end of the game. Complete list of ALL victory point / scoring sources. Final scoring procedure. Tiebreaker rules. |

**Additional subtopics:** Rogues may add game-specific subtopics to the Rules tab when a mechanic is complex enough to warrant its own section. Examples:
- Catan: add `"trading"` and `"development-cards"` subtopics
- Spirit Island: add `"fear"` and `"invader-phase"` subtopics
- Root: add separate subtopics per faction

Additional subtopics go AFTER `special-mechanics` and BEFORE `endgame`. The `endgame` subtopic is always last in the Rules tab.

### Strategy Tab

| ID | Title | Required | Description |
|----|-------|----------|-------------|
| `opening-priorities` | Opening Priorities | **YES** | Specific, actionable advice for the first few turns. Not "have a strategy" — instead, "prioritize getting a settlement on an ore hex because you'll need ore for cities." |
| `common-mistakes` | Common Mistakes | **YES** | 3–5 concrete mistakes new players make and WHY they're mistakes. Each mistake should be one the player can immediately avoid. |
| `key-decisions` | Key Decisions | NO | The 2–3 biggest strategic trade-offs in the game. Omit for very simple party games (Skull, Love Letter) where strategy is primarily social/bluffing rather than decision-tree based. |

## 3. Schema Validation Rules (Paladin's Updated Checklist)

| # | Check | Pass Criteria |
|---|-------|---------------|
| 1 | `game_id` matches filename | `catan.json` → `"game_id": "catan"` |
| 2 | `complexity` is valid | One of: `party`, `gateway`, `midweight`, `heavy` |
| 3 | `source_url` is valid | Points to official rules PDF or publisher rules page |
| 4 | `schema_version` is `"2.0"` | Must be the string `"2.0"` |
| 5 | All 3 tabs present | `tabs.setup`, `tabs.rules`, `tabs.strategy` all exist |
| 6 | Required subtopics present | See required subtopics table above per tab |
| 7 | No empty content | Every subtopic's `content` field is substantive (not placeholder text, not empty string) |
| 8 | Subtopic IDs are valid | Lowercase, hyphenated, unique within their tab. Required subtopics use exact standard IDs. |
| 9 | `endgame` is last in Rules | The `endgame` subtopic must be the final item in `tabs.rules.subtopics` |
| 10 | Player count accurate | Matches the game's published player count |
| 11 | Component list plausible | Correct types and counts of pieces, cards, dice for this game |
| 12 | Turn structure accurate | Correctly describes what happens on a turn/round |
| 13 | Scoring complete | All VP sources listed, end-game trigger correct, tiebreakers included |
| 14 | Strategy is actionable | Gives specific advice, not generic platitudes |
| 15 | Token count in range | 1,200–5,000 total. See complexity-based guidelines. |
| 16 | No hallucinations | No invented mechanics, no rules from other games |
| 17 | No copyright violation | Original teaching text, not copied rulebook text |

**Rejection protocol (unchanged):** A single failed check = rejection. Paladin specifies which check(s) failed and the exact correction needed. Vague rejections not permitted.

## 4. LLM Context Flattening (Backend Implementation)

The backend's `knowledge.py` service flattens all subtopics into a single Markdown string for the Q&A system prompt. Here is the exact algorithm:

```
For each tab in [setup, rules, strategy]:
    Emit "## {Tab Name}" (e.g., "## Setup")
    For each subtopic in tab.subtopics:
        Emit "### {subtopic.title}"
        Emit subtopic.content
        Emit blank line
```

**Example output for Catan (abbreviated):**

```markdown
## Setup

### What's in the Box
19 terrain hexes (4 forest, 4 pasture, 4 fields, 3 hills, 3 mountains, 1 desert)...

### Board Layout
1. Arrange the 6 sea frame pieces into a hexagonal border...

### Player Setup
Each player takes: 5 settlements, 4 cities, 15 roads...

### Starting the Game
Starting with a random first player and going clockwise...

## Rules

### Turn Structure
On your turn: 1) Roll dice for resource production, 2) Trade, 3) Build.

### Actions & Options
...

### Trading
...

### Development Cards
...

### Endgame & Scoring
First player to reach 10 victory points on their turn wins...

## Strategy

### Opening Priorities
Prioritize high-probability numbers (6, 8, 5, 9)...

### Common Mistakes
Don't hoard cards past 7...

### Key Decisions
Cities vs. expansion...
```

This flattened format becomes the `{KNOWLEDGE BASE}` block in the existing LLM system prompt from Section 12 of the Master Architecture doc. The system prompt itself does NOT change — only the content block inside it gets richer structure.

**Tab name mapping for the header:**
- `setup` → `"Setup"`
- `rules` → `"Rules"`
- `strategy` → `"Strategy"`

## 5. Token Budget (Updated)

| Complexity | Token Range | Typical Games |
|------------|-------------|---------------|
| `party` | 1,200 – 2,000 | Codenames, Skull, Love Letter, Coup |
| `gateway` | 2,000 – 3,500 | Catan, Ticket to Ride, Azul, Splendor |
| `midweight` | 2,500 – 4,000 | Wingspan, Everdell, Lords of Waterdeep |
| `heavy` | 3,500 – 5,000 | Brass: Birmingham, Spirit Island, Agricola |

**Hard floor:** 1,200 tokens. Below this, the content is too thin to teach the game.
**Hard ceiling:** 5,000 tokens. Above this, the Rogue is writing a rulebook, not a teaching guide.

**LLM context impact:** Negligible. GPT-5.3-Codex has a 128k context window. At 5,000 tokens per game, the system prompt + knowledge base + full conversation history is under 10k tokens. We have 120k of headroom. Token budget is about content quality, not model constraints.

## 7. The `extensions` Field

The `extensions` field is an empty object (`{}`) in v2.0. It exists as a reserved namespace for future enhancements without requiring a schema version bump. Planned (but NOT implemented tonight):

- `extensions.images` — Setup diagram URLs
- `extensions.bgg_id` — BoardGameGeek integration
- `extensions.expansion_ids` — Links to expansion knowledge bases
- `extensions.video_url` — Teaching video link

**Rogues: leave `extensions` as `{}`. Do not populate it. Paladin: ignore this field during validation.**

## 8. Updated LLM System Prompt

Minor update to the system prompt to reflect the structured content:

```
SYSTEM: You are GameMaster AI, a friendly and knowledgeable board game teacher
working at a board game cafe. You are currently teaching {game_title}.

Use ONLY the knowledge base below to answer questions. The knowledge base is
organized into Setup, Rules, and Strategy sections with labeled subtopics.
If the knowledge base does not contain the answer, say "I'm not sure about
that specific rule — you may want to check the rulebook for {game_title}."
NEVER invent or guess at rules.

Be concise. Players are at a table with the game in front of them — they
need quick, clear answers, not essays.

KNOWLEDGE BASE:
{flattened markdown from all tabs}
```

The `MODE` parameter from v1.0 is **removed** from the system prompt. Here's why: modes were a proxy for what the user wanted. Now, Tabs 1–3 handle setup/rules/strategy directly as pre-rendered content. The only time the LLM is invoked is Tab 4 (Q&A), which is always "answer the question as concisely and accurately as possible." There's no reason to mode-switch the LLM anymore.

If Tim wants to re-add modes later (e.g., a "explain like I'm 5" toggle), we can. But tonight, one mode. Keep it simple.
